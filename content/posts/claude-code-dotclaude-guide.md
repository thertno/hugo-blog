你有没有这种感觉：刚开始用 Claude Code 的时候，项目里就一个 `CLAUDE.md` 文件，清清爽爽的。结果用着用着，这个文件从 50 行变成了 200 行，后来又加了各种脚本、配置、指令……最后整个项目变得乱七八糟，连你自己都看不懂了。

别慌，这不是你的问题，是你还没学会"收拾"这些文件。

## 为什么要整理？

说白了，`.claude/` 文件夹就是 Claude Code 的"说明书"。说明书越清晰，AI 越知道该怎么干活。说明书乱成一团，AI 就会做出一些让你哭笑不得的事情。

整理好的 `.claude/` 文件夹有 3 个好处：

1. **AI 更聪明**：指令放对地方，它就能精准理解你的意图
2. **好维护**：过了三个月回来看，你还能看懂自己写了啥
3. **团队协作**：别人接手项目，一看就知道怎么指挥 Claude

## 核心原则：分清楚"谁干什么"

想象你在一家公司，你不可能让同一个人又当 HR 又当程序员还当保洁员对吧？`.claude/` 文件夹也是一样的道理，每种文件有它自己的职责。

### 规则一：顶层保持"苗条"

项目根目录只需要两个东西：

- **`CLAUDE.md`** —— 这是给 Claude 的"项目说明书"，告诉它这个项目是干啥的、用了什么技术栈、有什么基本的规矩
- **`CLAUDE.local.md`** —— 这是你**个人**的偏好设置，不纳入版本控制，只有你自己看

`.claude/settings.json` 管的是 Claude 在这项目里"怎么干活"（比如能不能联网、能不能执行脚本等权限控制）。

`CLAUDE.local.md` 和 `settings.local.json` 是你个人的"小抄"，不用提交到 git 里。这样既能保证团队有一致的基线配置，又能让每个人按照自己的习惯微调。

### 规则二：全局指导 vs 专项指导要分开

`CLAUDE.md` 放的是**任何时候都要遵守**的内容，比如：

- 项目用了什么技术栈（Vue + Node.js + MySQL 之类的）
- 整体架构是怎么回事（前端怎么调后端、数据怎么流转的）
- 最关键的命令是什么（怎么跑项目、怎么跑测试）
- 所有人都得遵守的代码规范

而那些**只在某些场景才用得到**的内容，就应该挪到 `rules/` 文件夹里：

```
.claude/
└── rules/
    ├── frontend.md      ← 前端相关的规矩
    ├── backend-api.md   ← 后端接口的规矩
    ├── testing.md       ← 测试怎么写的规矩
    └── data-pipelines.md ← 数据处理相关的规矩
```

**什么时候该拆分？** 当你发现 CLAUDE.md 已经长到翻好几屏才能看完的时候，就该把一些内容拆出去了。

### 规则三：自动执行的脚本 vs 手动用的工作流

`hooks/` 和 `commands/` 是两个特别容易搞混的文件夹：

**`hooks/` = 自动运行的脚本**

就像你设的闹钟，到点它就响了，不用你管。比如：

- 自动拦截危险的命令（`block-dangerous-commands.sh`）—— Claude 想删库的时候拦住它
- 自动格式化代码（`format-edits.sh`）—— Claude 改完代码自动排版
- 改完代码自动跑测试（`run-tests-before-stop.sh`）—— 确保改动不出问题

**`commands/` = 可复用的提示词工作流**

就像你保存的"快捷回复"，你想用的时候再调出来。比如：

- `review-pr.md` —— 审查 PR 的提示词
- `write-tests.md` —— 写测试的提示词
- `summarize-changes.md` —— 总结本次改动

总结一句话：**hooks 是自动的，commands 是手动的。**

### 规则四：更高级的打包方式

当你有一些特别复杂的任务，需要多步骤配合才能完成时，就需要用到 `skills/` 和 `agents/` 了。

**`skills/` 就像一个"工具包"**

里面不仅有说明文档，还有配套的模板和规则。比如一个"发布准备"的 Skill，可能包含：

- `SKILL.md` —— 说明这个技能怎么用
- `release-template.md` —— 发布时要填的模板

**`agents/` 就像"专人"**

每个 Agent 有自己专注的角色，比如：

- `code-reviewer.md` —— 专门负责代码审查
- `security-auditor.md` —— 专门负责安全检查
- `docs-writer.md` —— 专门负责写文档

Skills 和 Agents 的区别可以这样理解：**Skills 是"一套工具"，Agent 是"一个人"。**

### 规则五：团队的东西归团队，个人的东西归自己

```text
# 大家共享的（放项目里）
your-project/
├── CLAUDE.md                  ← 团队指令
└── .claude/
    ├── settings.json          ← 团队配置
    ├── rules/                 ← 团队规则
    └── hooks/                 ← 团队自动化

# 你自己用的（放 HOME 目录）
~/.claude/
├── CLAUDE.md                  ← 个人偏好
├── settings.json              ← 个人配置
├── skills/                    ← 个人技能
└── agents/                    ← 个人 Agent
```

判断标准就一个问题：**这个配置是不是为了让大家干活更一致？** 如果是，放项目里。如果是你自己的习惯，放 `~/.claude/` 里。

## 一步一步来，别一口吃成胖子

刚开始用 Claude Code，别上来就建一堆空文件夹。按这个顺序来：

1. **刚起步**：`CLAUDE.md` + `.claude/settings.json`，最基础的就够用了
2. **指令变多了**：加 `rules/`，把专项指导拆出去
3. **想自动做事了**：加 `hooks/`，让 Claude 自动帮你干活
4. **提示词开始重复了**：加 `commands/`，把常用的提示词存起来
5. **工作流变复杂了**：加 `skills/`，打包成完整的工作流
6. **需要专人负责了**：加 `agents/`，给不同角色建专门的 Agent

## 新手常踩的坑

| ❌ 错误做法 | ✅ 正确做法 |
|------------|------------|
| CLAUDE.md 塞了 500 行 | 专项内容拆到 `rules/` 里 |
| 项目刚建就用上所有文件夹 | 等真需要了再加 |
| 个人偏好混到团队配置文件里 | 用 `CLAUDE.local.md` 或 `~/.claude/` |
| 文件名叫 `script1.sh` | 名字要一眼看出它是干嘛的 |
| 废弃了的文件不清理 | 定期打扫，删掉不用的文件 |

## 记住最后一句话

> 最高效的 `.claude/` 文件夹不是功能最多的，而是每个文件都知道自己为什么在那里的。

当你的 `.claude/` 井井有条时，Claude Code 就像一个训练有素的助手，你说什么它都能秒懂，而不是在那里"自由发挥"。
