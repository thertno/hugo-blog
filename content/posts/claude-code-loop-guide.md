你有没有这种感觉：用 Claude Code 写代码，改一丢丢，看结果，不满意，再改一丢丢……整个过程就像在当「人肉遥控器」。

Claude Code 团队最近发了篇博客，提出一个观点：**AI 编程的核心操作单元，正在从「一次对话」变成「一个循环」**。

这个循环，就是 Loop。

## 什么是 Loop？

一句话：**你定好目标和结束条件，AI 自己反复跑，跑完才交活。**

以前是你一条条下指令，AI 做一步停一步。现在是你告诉 AI「做到什么算完」，AI 自己读代码、改代码、跑测试、检查，不满意就再来一轮，直到达标。

## 四种 Loop，四种「放手程度」

### 1. 回合制 Loop（最基础）

就是你现在的用法：你发一条，AI 改完交回来，你检查，再发下一条。

**关键技巧**：别让 AI 改完就说「完成了」。用 SKILL.md 把你的验证步骤写进去——比如改了前端，必须启动 dev server、截图对比、检查控制台没报错，才算交活。

这是后面所有高级 Loop 的基础。如果 AI 连自我验证都做不好，给它再多自主权也白搭。

### 2. 目标制 Loop（推荐先学这个）

用 `/goal` 命令定义成功标准，AI 会反复迭代直到达标。

比如：

```
/goal 把首页 Lighthouse 分数提到 90 以上，最多试 5 次
```

**核心区别**：回合制里，AI 做完一步就想停（因为它不知道什么算「够好」）。`/goal` 把「完成标准」说清楚了，AI 就不用猜。

**经验**：目标越量化，效率越高。「优化一下性能」是模糊目标，AI 跑两圈就停。「LCP 降到 2.5 秒以下」是精确目标，AI 会持续迭代直到命中。

### 3. 定时制 Loop（适合重复工作）

`/loop` 按时间间隔重复执行同一个 prompt，`/schedule` 搬到云端，你关机也照跑。

比如：

```
/loop 5m 检查我的 PR，处理 review 评论，修复失败的 CI
```

适合：每天总结 Slack 消息、每隔几分钟检查 CI、每周生成依赖更新报告。人做这些是「想起来就去看一眼」，定时 Loop 把它变成自动的。

### 4. 主动式 Loop（最激进）

没有人类实时参与，AI 自己监听事件、自己决定行动、自己验证结果。

比如：每小时检查反馈渠道 → 每个 bug 必须被分类、修复、回复 → 多个 Agent 并行探索方案 → 全程无需人工审批。

**一句话**：触发、执行、验证、交付，全自动。

## 怎么选？一张图搞定

| 你的情况 | 用哪种 |
|---------|--------|
| 在探索、试验 | 回合制，重点写好验证 Skill |
| 你知道「做完」长什么样 | `/goal`，退出条件量化 |
| 这件事每天/每周重复做 | `/loop` 或 `/schedule` |
| 输入输出都能标准化 | 上 Proactive，全自动 |

**顺序很重要**：别跳过基础直接上 Proactive。如果回合制的验证 Skill 都没写好，自动循环只会更快地产出垃圾。

## 一个容易忽略的问题：Token 成本

Loop 越复杂，Token 消耗越大。Dynamic Workflows 可以并行派出几百个 AI Agent，一次 run 烧掉的 Token 可能让你心痛。

**建议**：先跑一个小切片，确认成本和质量可控，再放大。

## 一句话总结

从 prompt 到 loop，本质上是一个**权力转移**的过程：你把判断权和执行权交给 AI，自己退到更上游——设计循环结构、定义成功标准、构建验证机制。

不是每个任务都需要 loop，但如果你还在逐条 prompt 驱动一个重复性的工作流，你正在做一件自动化该做的事。
